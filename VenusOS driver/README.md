## Overview

This driver connects a **TeslaBMS-ESP32** battery management system to a **Victron Venus OS** GX device (Cerbo GX, Raspberry Pi GX, CCGX, etc.) over a USB serial connection.

Once running, Venus OS treats the Tesla pack as a fully managed battery. The Victron DVCC system automatically reads charge voltage limits (CVL), charge current limits (CCL), and discharge current limits (DCL) from the driver and distributes them to every connected Victron device (MPPT chargers, Multis, Quattros) without any manual configuration.

```
┌────────────────────────────────────────────────────────┐
│                    Venus OS (Cerbo GX)                  │
│                                                         │
│  ┌──────────────┐  DVCC  ┌────────────────────────┐   │
│  │ MPPT / Multi │◄──────►│  dbus-systemcalc-py    │   │
│  │  (charger)   │        │  (Victron's controller) │   │
│  └──────────────┘        └───────────┬────────────┘   │
│                                      │ reads D-Bus      │
│                           ┌──────────▼──────────────┐  │
│                           │    dbus-teslabms.py      │  │
│                           │    (this driver)         │  │
│                           └──────────▲──────────────┘  │
│                                      │ USB serial        │
└──────────────────────────────────────┼─────────────────┘
                                       │
                            ┌──────────▼──────────┐
                            │   ESP32 TeslaBMS    │
                            │  (your firmware)    │
                            └─────────────────────┘
```

## Hardware Requirements

| Item | Notes |
|---|---|
| Victron GX device | Cerbo GX, Raspberry Pi running Venus OS, CCGX, or similar |
| Venus OS version | **v3.00 or newer** (uses `gi.repository.GLib`) |
| USB → UART adapter | CP2102, CH340, or similar; connects ESP32 to GX device |
| ESP32 TeslaBMS | Running firmware with `ExternalCommsLayer` at 115200 baud |

The driver auto-detects the serial port by scanning `/dev/ttyUSB*` and `/dev/ttyACM*` and sending a test frame. No manual port configuration is required.

## Installation

### Step 1 — Copy files to the GX device

Transfer `dbus-teslabms.py` and `install.sh` to the GX device via SCP:

```bash
scp dbus-teslabms.py install.sh root@<GX-IP-ADDRESS>:/tmp/
```

Or copy them via USB stick to `/tmp/`.

### Step 2 — Run the installer

```bash
ssh root@<GX-IP-ADDRESS>
cd /tmp
bash install.sh
```

The installer will:
1. Verify Venus OS ≥ v3.00
2. Copy `dbus-teslabms.py` to `/data/etc/dbus-teslabms/`
3. Create a runit service at `/service/dbus-teslabms/`
4. Add a hook to `/data/rc.local` so the service survives firmware updates
5. Start the driver immediately

### Step 3 — Verify it is running

```bash
# Check runit service status
svstat /service/dbus-teslabms

# Watch the live log
tail -f /var/log/dbus-teslabms/current

# Confirm the D-Bus service is registered
dbus -y com.victronenergy.battery.teslabms / GetValue
```

### Uninstall

```bash
bash /data/etc/dbus-teslabms/uninstall.sh
```

## How It Works

### Driver startup sequence

```
1.  rc.local calls enable.sh on every boot
2.  enable.sh creates /service/dbus-teslabms/
3.  runit starts dbus-teslabms.py automatically
4.  DBusGMainLoop initialised
5.  TeslaBMSSerial.start() → background reader thread begins
        a. Scans /dev/ttyUSB* and /dev/ttyACM* for ESP32
        b. Sends test frame; validates 27-byte reply
        c. Opens confirmed port at 115200 baud
        d. Listens for unsolicited 27-byte heartbeat frames
6.  VeDbusService registered as com.victronenergy.battery.teslabms
7.  GLib timer fires every 1000 ms → publish() writes D-Bus paths
8.  dbus-systemcalc-py detects the battery service automatically
9.  DVCC distributes CVL/CCL/DCL to all Victron chargers/inverters
```

### Steady-state data flow

```
ESP32 firmware (every ~1000 ms)
  └─ sends 27-byte frame unsolicited
        │
        ▼
  reader thread (background)
  _parse_frame()
  └─ updates bms.voltage, current, soc, temp,
     alarm_flags, overlord_state,
     eep_overvoltage, eep_undervoltage,
     eep_overtemp, eep_undertemp,
     eep_num_modules, eep_num_strings,
     cell_count, capacity_ah
        │
        ▼ (attributes read by main thread every 1 s)
  GLib timer → publish()
  └─ writes all D-Bus paths
        │
        ▼
  dbus-systemcalc-py
  └─ instructs MPPT / Multiplus via VE.Bus / VE.Direct
```

### Stale frame detection

If no frame is received for **2.5 seconds** the driver sends `EXT_CMD_SEND_DATA (0x03)` to demand an immediate reply. If no frame arrives for **15 seconds** the BMS is declared offline:

- `/Connected` → 0
- `/Alarms/BmsCable` → 1 (Warning)
- `/State` → 0 (Initialising)
- DVCC falls back to its own defaults

### Auto-reconnect

If the serial port is lost (USB disconnected, ESP32 reboot) the reader thread closes the port and re-runs the auto-detection scan every 5 seconds until the ESP32 is found again. No driver restart is required.

### EEPROM config — no persistence needed

Every 27-byte heartbeat frame carries the current ESP32 EEPROM values alongside the live telemetry. Venus OS always reflects exactly what is in EEPROM. If a user changes a threshold in the BMS configurator it appears in Venus OS within one heartbeat interval (~1 second). There is no caching, no stale settings file, and no separate config request.

## Frame Protocol

### Heartbeat / data frame (ESP32 → Venus OS, every ~1 s)

**27 bytes total:** `[0xAA]` + 24-byte payload + 2-byte CRC

| Bytes | Field | Type | Encoding | Unit |
|---|---|---|---|---|
| 0 | Start byte | — | `0xAA` | — |
| 1–2 | Pack voltage | uint16 BE | ÷ 100 | V (10 mV res) |
| 3–4 | Pack current | int16 BE | ÷ 10 | A (100 mA res, + = charge) |
| 5 | SOC | uint8 | raw | % |
| 6 | Temperature | int8 | raw | °C (signed) |
| 7–8 | Power | int16 BE | raw | W |
| 9–10 | Avg cell voltage | uint16 BE | ÷ 100 | V (10 mV res) |
| 11–12 | Alarm flags | uint16 BE | bitmask | see below |
| 13 | Overlord state | uint8 | enum | 0/1/2 |
| 14 | Contactor state | uint8 | enum | raw |
| 15–16 | EEPROM over-voltage | uint16 BE | ÷ 1000 | V (1 mV res) |
| 17–18 | EEPROM under-voltage | uint16 BE | ÷ 1000 | V (1 mV res) |
| 19 | EEPROM over-temp | int8 | raw | °C (signed) |
| 20 | EEPROM under-temp | int8 | raw | °C (signed) |
| 21 | Number of modules | uint8 | raw | count |
| 22 | Parallel strings | uint8 | raw | count |
| 23 | Reserved | uint8 | `0x00` | — |
| 24 | Reserved | uint8 | `0x00` | — |
| 25–26 | CRC-16/MODBUS | uint16 LE | over bytes 1–24 | — |

### Command frame (Venus OS → ESP32, on demand)

**4 bytes total:** `[0xAA]` + 1-byte command + 2-byte CRC

| Bytes | Field | Notes |
|---|---|---|
| 0 | Start byte `0xAA` | |
| 1 | Command code | See command table below |
| 2–3 | CRC-16/MODBUS LE | Computed over byte 1 only |

### CRC algorithm

CRC-16/MODBUS: initial value `0xFFFF`, polynomial `0xA001` (reflected).  
Computed over the **payload bytes only** (start byte excluded).

## Alarm & Fault Reference

### Alarm flags bitmask (payload bytes 11–12)

| Bit | Constant | Meaning | Victron path | Level |
|---|---|---|---|---|
| 0 | `ALARM_OVER_VOLTAGE` | Cell over-voltage | `/Alarms/HighVoltage`, `/Alarms/HighCellVoltage` | Alarm (2) |
| 1 | `ALARM_UNDER_VOLTAGE` | Cell under-voltage | `/Alarms/LowVoltage`, `/Alarms/LowCellVoltage` | Alarm (2) |
| 2 | `ALARM_OVER_TEMP` | Over-temperature | `/Alarms/HighTemperature` | Alarm (2) |
| 3 | `ALARM_UNDER_TEMP` | Under-temperature | `/Alarms/LowTemperature` | Alarm (2) |
| 4 | `ALARM_OVER_CURRENT` | Over-current | `/Alarms/HighChargeCurrent`, `/Alarms/HighDischargeCurrent` | Alarm (2) |
| 5–15 | — | **Reserved / not yet implemented** | — | — |

> **Note:** Alarm bits 5–15 are reserved. As the ESP32 Overlord exposes additional fault bits, add them to `ALARM_*` constants in both `ExternalCommsLayer.h` and `dbus-teslabms.py`.

### Victron alarm levels

| Value | Meaning |
|---|---|
| 0 | OK — no alarm |
| 1 | Warning — attention needed |
| 2 | Alarm — action required |

### Overlord state (payload byte 13)

| Value | Meaning | Driver behaviour |
|---|---|---|
| 0 | Normal | Charge and discharge allowed. `/State` = 9 (Running) |
| 1 | Fault / Shutdown | Contactors open. CCL and DCL set to 0. `/State` = 10 (Error). `/Alarms/InternalFailure` = 2 |
| 2 | Storage mode | Charge and discharge allowed at reduced limits (future). `/Info/ChargeMode` = "Storage" |

### SOC-based low SOC alarm

These are generated by the driver from the SOC field, independent of the ESP32 alarm flags:

| SOC | `/Alarms/LowSoc` |
|---|---|
| ≥ 10 % | 0 (OK) |
| 5–9 % | 1 (Warning) |
| < 5 % | 2 (Alarm) |

### BMS cable alarm

Generated by the driver from frame reception timing, not from the ESP32:

| Condition | `/Alarms/BmsCable` |
|---|---|
| Frame received within 15 s | 0 (OK) |
| No frame for > 15 s | 1 (Warning) |

### What is NOT yet implemented

| Feature | Status |
|---|---|
| Per-cell voltage alarms | Not implemented — ESP32 sends average cell voltage only. Per-cell data would require extending the frame protocol. |
| Cell imbalance alarm | Not implemented — requires per-cell data. |
| High internal temperature | Not implemented — no MOS/FET temperature sensor in current firmware. |
| Fuse blown alarm | Not implemented. |
| State of Health (SOH) | Not implemented — `/Soh` published as `None`. |
| Charge cycle counting | Not implemented — `/History/ChargeCycles` published as `None`. |
| Energy throughput (Wh) | Not implemented — `/History/ChargedEnergy` / `DischargedEnergy` published as `None`. |

## Commands

Commands are sent by writing `1` to a writable D-Bus path on the battery service. The driver transmits the corresponding 4-byte command frame to the ESP32 and resets the path back to `0` immediately.

The ESP32 always responds to a command with a fresh 27-byte data frame.

### Available commands

| D-Bus path | Command sent | ESP32 action |
|---|---|---|
| `/Control/Shutdown` | `EXT_CMD_SHUTDOWN (0x01)` | Calls `Overlord.requestShutdown()` → opens contactors |
| `/Control/Startup` | `EXT_CMD_STARTUP (0x02)` | Calls `Overlord.requestStartup()` → closes contactors |

### Method 1 — Command line (SSH)

Available immediately, no extra software required. Useful for testing and scripting.

```bash
# Open SSH session to GX device
ssh root@<GX-IP-ADDRESS>

# Shutdown (open contactors)
dbus -y com.victronenergy.battery.teslabms /Control/Shutdown SetValue %1

# Startup (close contactors)
dbus -y com.victronenergy.battery.teslabms /Control/Startup SetValue %1

# Read current SOC
dbus -y com.victronenergy.battery.teslabms /Soc GetValue

# Read current EEPROM overvoltage threshold
dbus -y com.victronenergy.battery.teslabms /Config/CellOverVoltage GetValue
```

### Method 2 — Node-RED (recommended for automation)

Node-RED runs natively on Venus OS Large image. It includes a built-in D-Bus node.

**Install Venus OS Large** if not already:  
https://github.com/victronenergy/venus/wiki/large-image

**Example: manual shutdown button flow**

```
[Button inject]  →  [Set msg.payload = 1]  →  [D-Bus out]
                                                  service: com.victronenergy.battery.teslabms
                                                  path:    /Control/Shutdown
```

**Example: automatic low-SOC shutdown**

```
[D-Bus in]              [Function node]          [D-Bus out]
service: ...teslabms  → if msg.payload < 5   →   /Control/Shutdown
path: /Soc              then msg.payload = 1
                        else return null
```

**Example: read live EEPROM config**

```
[D-Bus in]  →  [Debug]
service: com.victronenergy.battery.teslabms
path:    /Config/CellOverVoltage
```

### Method 3 — Venus OS GUI button (advanced)

The Venus OS QML GUI can be patched to add a custom settings page with Shutdown / Startup buttons. This requires QML development and use of the `custom-gui-install.sh` mechanism from the mr-manuel dbus-serialbattery project.

> This is not included in the current driver version. Raise a feature request if needed.

### Method 4 — HTTP / MQTT via VRM (remote)

If the GX device is connected to VRM (Victron Remote Monitoring), D-Bus paths can be written remotely via the VRM MQTT broker:

```
Topic:  W/<VRM-portal-ID>/battery/<device-instance>/Control/Shutdown
Value:  {"value": 1}
```

Refer to the Victron MQTT documentation:  
https://github.com/victronenergy/dbus-mqtt

## D-Bus Path Reference

Service name: `com.victronenergy.battery.teslabms`

### Identity

| Path | Type | Value |
|---|---|---|
| `/ProductName` | str | `TeslaBMS-ESP32` |
| `/FirmwareVersion` | str | `1.2` |
| `/HardwareVersion` | str | `ESP32 HEX v1.0` |
| `/Connected` | int | 0 = offline, 1 = online |
| `/CustomName` | str | User-editable label |
| `/State` | int | 0=Init 9=Running 10=Error 14=Standby |

### Live telemetry

| Path | Unit | Notes |
|---|---|---|
| `/Soc` | % | 0–100 |
| `/Dc/0/Voltage` | V | Pack voltage |
| `/Dc/0/Current` | A | Positive = charging |
| `/Dc/0/Power` | W | |
| `/Dc/0/Temperature` | °C | Average pack temperature |
| `/System/MinCellVoltage` | V | Average cell voltage (per-cell not available) |
| `/System/MaxCellVoltage` | V | Same as Min — average only |
| `/System/MinCellTemperature` | °C | |
| `/System/MaxCellTemperature` | °C | |
| `/System/NrOfCellsPerBattery` | int | Derived: modules ÷ strings |

### DVCC limits (read by dbus-systemcalc-py)

| Path | Unit | Notes |
|---|---|---|
| `/Info/MaxChargeVoltage` | V | CVL = (EEPROM OV − 0.10 V) × cell count |
| `/Info/BatteryLowVoltage` | V | EEPROM UV × cell count |
| `/Info/MaxChargeCurrent` | A | 200 A, or 0 if fault |
| `/Info/MaxDischargeCurrent` | A | 200 A, or 0 if fault |
| `/Info/ChargeMode` | str | Normal / Storage / Fault |

### Capacity

| Path | Unit | Notes |
|---|---|---|
| `/InstalledCapacity` | Ah | (modules ÷ strings) × 22 Ah |
| `/Capacity` | Ah | Remaining = installed × SOC% |
| `/ConsumedAmphours` | Ah | Negative convention (Victron standard) |

### IO / FET flags

| Path | Value | Notes |
|---|---|---|
| `/Io/AllowToCharge` | 0 or 1 | 0 when overlord in Fault |
| `/Io/AllowToDischarge` | 0 or 1 | 0 when overlord in Fault |

### Alarms

| Path | Levels | Trigger |
|---|---|---|
| `/Alarms/LowVoltage` | 0/2 | `ALARM_UNDER_VOLTAGE` bit |
| `/Alarms/HighVoltage` | 0/2 | `ALARM_OVER_VOLTAGE` bit |
| `/Alarms/LowCellVoltage` | 0/2 | `ALARM_UNDER_VOLTAGE` bit |
| `/Alarms/HighCellVoltage` | 0/2 | `ALARM_OVER_VOLTAGE` bit |
| `/Alarms/LowSoc` | 0/1/2 | SOC < 10% warning, < 5% alarm |
| `/Alarms/HighTemperature` | 0/2 | `ALARM_OVER_TEMP` bit |
| `/Alarms/LowTemperature` | 0/2 | `ALARM_UNDER_TEMP` bit |
| `/Alarms/HighChargeCurrent` | 0/2 | `ALARM_OVER_CURRENT` bit |
| `/Alarms/HighDischargeCurrent` | 0/2 | `ALARM_OVER_CURRENT` bit |
| `/Alarms/InternalFailure` | 0/2 | Overlord state = Fault (1) |
| `/Alarms/BmsCable` | 0/1 | No frame for > 15 s |

### Live EEPROM config mirror (read-only)

Updated every heartbeat frame. Always reflects current ESP32 EEPROM values.

| Path | Unit | Notes |
|---|---|---|
| `/Config/CellOverVoltage` | V | EEPROM OV threshold per cell |
| `/Config/CellUnderVoltage` | V | EEPROM UV threshold per cell |
| `/Config/CellFloatVoltage` | V | OV − 0.10 V (Venus OS calculated) |
| `/Config/OverTemp` | °C | EEPROM OT threshold |
| `/Config/UnderTemp` | °C | EEPROM UT threshold |
| `/Config/NumModules` | int | Total modules detected by BMS scan |
| `/Config/NumStrings` | int | Parallel string count from EEPROM |
| `/Config/CellCount` | int | Series cells = modules ÷ strings |
| `/Config/CapacityAh` | Ah | (modules ÷ strings) × 22 Ah |

### Commands (writable)

| Path | Write | Action |
|---|---|---|
| `/Control/Shutdown` | `1` | Sends `EXT_CMD_SHUTDOWN` → Overlord opens contactors |
| `/Control/Startup` | `1` | Sends `EXT_CMD_STARTUP` → Overlord closes contactors |

## Known Limitations

| Limitation | Detail |
|---|---|
| Average cell voltage only | The frame protocol transmits one average cell voltage. Per-cell monitoring requires extending the frame format on the ESP32 side. |
| No SOH | State of Health is not calculated. `/Soh` is always `None`. |
| No energy history | Charged/discharged energy counters are not implemented. |
| CCL/DCL are static | Max charge and discharge current limits are hardcoded in the driver (`MAX_CHARGE_CURRENT`, `MAX_DISCHARGE_CURRENT`). They do not derate with temperature or SOC. Implementing dynamic derating requires extending the alarm flags or adding new fields to the frame. |
| Float offset is fixed | The CVL float offset (`CELL_FLOAT_OFFSET = 0.10 V`) is a compile-time constant in the Python driver. It is not configurable from the Venus OS GUI in the current version. |
| Single pack only | The driver supports one ESP32 on one serial port. Multiple packs are not supported in the current version. |
| No GUI settings page | There is no custom Venus OS GUI page. Settings are read-only mirrors of ESP32 EEPROM. Commands are issued via CLI, Node-RED, or MQTT. |

## Troubleshooting

### Driver not starting

```bash
# Check runit sees the service
ls /service/dbus-teslabms

# If missing, re-run the enable script
bash /data/etc/dbus-teslabms/enable.sh

# Check the log for Python errors
tail -50 /var/log/dbus-teslabms/current
```

### ESP32 not detected

```bash
# Check that the USB adapter is visible to the OS
ls /dev/ttyUSB* /dev/ttyACM*

# Check dmesg for USB events
dmesg | tail -20

# Manually probe the port (replace ttyUSB0 as needed)
python3 -c "
import serial, struct, time
def crc(d):
    c=0xFFFF
    for b in d:
        c^=b
        for _ in range(8): c=(c>>1)^0xA001 if c&1 else c>>1
    return c
s=serial.Serial('/dev/ttyUSB0',115200,timeout=1)
cb=bytes([0x03])
s.write(b'\xAA'+cb+struct.pack('<H',crc(cb)))
time.sleep(0.4)
d=s.read(27)
print('Received',len(d),'bytes:', d.hex())
s.close()
"
```

### Venus OS not showing the battery

```bash
# Confirm the D-Bus service is registered
dbus -y com.victronenergy.battery.teslabms /Connected GetValue
# Expected: 1

# If command fails, the service is not running — check the log
tail -f /var/log/dbus-teslabms/current
```

### DVCC not controlling the charger

1. In the Venus OS GUI go to **Settings → System Setup → DVCC** and confirm DVCC is **enabled**.
2. Confirm `/Connected` = 1 and `/Info/MaxChargeVoltage` has a non-zero value.
3. Check that the MPPT/Multi firmware supports DVCC (most do from 2018 onward).

### Values look wrong (voltage off by 10×)

This would indicate a frame format mismatch between the ESP32 firmware and the Python driver. Verify that both sides use the **27-byte frame** defined in this document. Older 21-byte frames are not compatible.

### Driver crashes in a loop

```bash
# View recent crash output
cat /var/log/dbus-teslabms/current | tail -100

# Common causes:
# - velib_python not found (install dbus-serialbattery first)
# - Wrong Venus OS version (need v3.x for gi.repository.GLib)
# - Permission error on /dev/ttyUSB0 (should not happen as root)
```
